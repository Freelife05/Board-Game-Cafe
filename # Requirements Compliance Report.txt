## 3. Security – Backend

**JWT Bearer token authentication with role-based authorization.**

### How it works

1. A client sends credentials to `POST /api/auth/login`.
2. `AuthService.LoginAsync` verifies the BCrypt-hashed password and returns a signed JWT.
3. The token contains claims: `NameIdentifier` (user id), `Name`, `Email`, `Role`, expiry 8 hours.
4. Every subsequent request must include `Authorization: Bearer <token>`.
5. ASP.NET Core validates the token automatically before the controller runs.

### Where it is configured

- **JWT setup**: `BoardGameCafe.API/Program.cs` lines 23–37
- **Token generation**: `BoardGameCafe.API/Services/AuthService.cs` — `GenerateToken` method
- **Password hashing**: `AuthService.cs` — `RegisterAsync` uses `BCrypt.HashPassword`, `LoginAsync` uses `BCrypt.Verify`
- **Authorize attributes**:
  - `[Authorize]` on the controller class protects the whole controller.
  - `[AllowAnonymous]` on GET endpoints allows public browsing.
  - `[Authorize(Roles = "Admin")]` on Create / Update / Delete restricts mutations to admins.

## 4. Security – Frontend

**Protected routes redirect unauthenticated users; every API request automatically carries the JWT.**

### How it works

1. On login the token and user object are stored in `localStorage`.
2. An Axios request interceptor attaches `Authorization: Bearer <token>` to every outgoing call.
3. A response interceptor listens for `401 Unauthorized`; when it fires it clears auth state and redirects to `/login`.
4. `ProtectedRoute` wraps sensitive pages — if no user is found in context, it redirects to `/login` before rendering.
5. Admin-only UI (create/edit/delete buttons) is conditionally rendered based on the `isAdmin` flag from `AuthContext`.

### Where it is implemented

| Concern | File |
|---|---|
| Token storage & logout | `boardgamecafe-client/src/contexts/AuthContext.tsx` |
| Request / response interceptors | `boardgamecafe-client/src/services/api.ts` |
| Protected route wrapper | `boardgamecafe-client/src/App.tsx` — `ProtectedRoute` component |
| Admin UI gating | `GamesPage.tsx`, `TablesPage.tsx`, `ReservationsPage.tsx` |

## 8. Global Exception Handling

**A middleware catches all unhandled exceptions and returns RFC 7807 Problem Details — no stack traces reach the client.**

### Response format (RFC 7807)

```json
{
  "type": "https://tools.ietf.org/html/rfc7807",
  "title": "Not Found",
  "status": 404,
  "detail": "Board game with id 99 was not found.",
  "instance": "/api/boardgames/99"
}
```

### Exception-to-status mapping

| Exception type | HTTP status |
|---|---|
| `KeyNotFoundException` | 404 Not Found |
| `UnauthorizedAccessException` | 401 Unauthorized |
| `InvalidOperationException` | 409 Conflict |
| `ArgumentException` | 400 Bad Request |
| Any other | 500 Internal Server Error |

### Where it is implemented

- **Middleware class**: `BoardGameCafe.API/Middleware/GlobalExceptionHandlingMiddleware.cs`
- **Registered in pipeline**: `BoardGameCafe.API/Program.cs` line 78 — `app.UseMiddleware<GlobalExceptionHandlingMiddleware>()` (placed before all other middleware so it wraps every request)
- **Logging**: The full exception (including stack trace) is written to the server log via `ILogger`; only the `Message` is sent to the client.

## 9. Async / Await

**All database queries, file I/O, and external communications use `async` / `await` throughout the stack.**

### Backend

Every service method is declared `async Task<T>` and awaits EF Core async operations:

| Operation | EF Core async method used |
|---|---|
| Check existence | `AnyAsync` |
| Fetch single record | `FindAsync`, `FirstOrDefaultAsync` |
| Fetch list | `ToListAsync` |
| Count for pagination | `CountAsync` |
| Persist changes | `SaveChangesAsync` |

All controller action methods are `async Task<IActionResult>` and `await` the service calls.

On startup, `Program.cs` awaits both `context.Database.MigrateAsync()` and `DbSeeder.SeedAsync(context)`.

### Frontend

All API calls use `async` arrow functions with `await api.get(...)` / `await api.post(...)` etc., wrapped in `try/catch` blocks. Examples:

- `GamesPage.tsx` — `fetchGames`, `handleSubmit`, `handleDelete`
- `TablesPage.tsx` — `fetchTables`, `handleSubmit`, `handleDelete`
- `ReservationsPage.tsx` — `fetchReservations`, `handleUpdate`, `handleDelete`
- `AuthContext.tsx` — `login`, `register`
